        HTML 템플릿을 그대로 쓰는데도 깨지거나 그런것이 전혀 없고 잘 열린다.
        웹브라우저는 TH문법이런것 모른다!

        JSP쓰면 완전히 깨지는데 타임리프는 VIEW를 HTML모양을 잘 살리면서 랜더링될때만 치환한다.

        TH:XXX가 붙은 부분은 서버사이드에서 렌더링이 되고 기존것을 대체한다 TH문법이 없으면 기존의것 사용하는것
        랜더링이 되면 TH문법 붙은 부분이 실행 된다.

        (1)링크표현할때는 th:href="@{ ~ / ~ } 이런식으로 사용

        리터럴 대체 문법 : | ... |
        문자와 표현식등은 분리되어 있기 떄문에 더해서 사용해야한다! 프린트문 찍을때 + 해주듯이
        그런데 리터럴 문법 사용하면 더하기 없이 편리하게 사용가능하다!

        반복할땐 th:each 기존에 사용하던것들과 방법유사

        변수표현 ${...}

        내용변경 th:text 내용의 값을 th:text의 값으로 변경한다. 기존의 값은 무시된다.

        url링크 표현식2
        th:href="@{/basic/items/{itemId}(itemId=${item.id}, query='test')}"
        앞에 ${}는 값으로 치환되어서 들어가고 뒤에 쿼리는 쿼리파라미터가 된다.
        http://localhost:8080/basic/items/1?query=test 이런식으로

    상품 수정
        HTML Form은 PUT,PATCH를 지원하지 않는다. GET,POST만 가능하다. 왜???

        PUT, PATCH는 HTTP API 전송시에 사용
        왜 등록할때는 REDIRECT안쓰고 수정할때만 REDIRECT 사용???
        중복 처리 방지를 위해서!!!

    PRG POST/REDIRECT/GET
        웹 브라우저의 새로 고침은 마지막에 서버에 전송한 데이터를 다시 전송하는것!
        새로고침하면서 같은 데이터가 계속 쌓이는것 어떻게 해결하나??

        POST요청에 REDIRECT로 응답해준다. 그럼 GET을 호출하는걸로 끝나게 된다!

        그래서 PRG 라고도 부른다.

    정리
        요구사항 분석은 보통 도메인 모델, 관리기능 으로 분석
        타임리프의 특징은 태그를 유지한상태로 속성만 바꾸는 역할

            핵심은 서버사이드에서 렌더링 되고, 기존것을 대체한다.

            html 파일을 직접 열어도 th 속성은 몰라서 무시 그래서 파일보기 유지하면서 템플릿기능도 수행 가능

        @ModelAttribute 생략가능 생략하면 타입의 첫글자 소문자로바꾼뒤에 모델에 담고 생성해준다. 기능2가지!

        POST에서 요청을 끝내지 않고 GET요청으로 끝내야한다! 데이터 중복 방지! PRG패턴!
